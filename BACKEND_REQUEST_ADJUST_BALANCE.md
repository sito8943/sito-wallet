# Backend Request: Adjust Account Balance

## Feature

Allow users to manually adjust the current balance of an account. This covers cases like:

- Correcting discrepancies between real and tracked balance
- Initial migration adjustments
- Cash accounts where not every transaction is tracked

---

## Prerequisite: Rename `initial` → `auto`

Before implementing the adjust-balance endpoint, rename the `initial` boolean flag to `auto` on both `Transaction` and `TransactionCategory` entities. This better reflects the flag's purpose (marking auto-generated transactions/categories, not just initial ones).

### Migration strategy

- Add a **Flyway/Liquibase migration** that renames the column in both tables (`initial` → `auto`) so existing data is preserved.
- Update Java entities, DTOs, mappers, specifications, and tests to use `auto` instead of `initial`.
- Frontend must also update to read `auto` instead of `initial`.

---

## Endpoint

```
PATCH /accounts/{accountId}/adjust-balance
```

### Request body

```json
{
  "newBalance": 1500.0,
  "description": "Optional text for the auto-generated transaction"
}
```

| Field         | Type     | Required | Description                                                                               |
| ------------- | -------- | -------- | ----------------------------------------------------------------------------------------- |
| `newBalance`  | `number` | Yes      | The absolute final balance the account should have                                        |
| `description` | `string` | No       | Description for the adjustment transaction. Defaults to `"Ajuste manual"` if not provided |

### Response

Return the **created transaction** as a full `TransactionDTO`:

```json
{
  "id": 42,
  "amount": 1500.00,
  "date": "2026-03-18T12:00:00",
  "description": "Ajuste manual",
  "auto": true,
  "category": { ... },
  "account": { ... }
}
```

---

## Backend behavior

### 1. Auto-create adjustment transaction

When the balance is adjusted, the backend **must** auto-create a transaction:

- **Amount** = `newBalance` (the absolute value the user entered, NOT a delta)
- **Balance update** — set `account.setBalance(newBalance)` directly, do NOT use `AccountBalanceUpdater` or calculate deltas
- **`auto: true`** — mark the transaction with the `auto` boolean flag (previously named `initial`)
- **Transaction category** — use an auto-generated category (`auto: true` on the category) similar to how initial balance categories work
- **Currency** — use the account's currency (`account.currency`)
- **Date** — `LocalDateTime.now()` (the moment of adjustment)
- **Description** — if the user provides `description`, use it; otherwise default to `"Ajuste manual"`
- **Visible** — the transaction should be visible in the transaction list (the frontend handles `auto: true` display with the autogenerated badge)

### 2. Ownership validation

Already handled by the existing pattern: `CurrentUserProvider` + `AccountSpecifications.fromUser(userId)` ensures the account belongs to the authenticated user.

### 3. Balance validation

- Respect the existing `balanceGreaterThanZero` feature flag
- If `balanceGreaterThanZero` is enabled and `newBalance < 0`, reject with appropriate error

### 4. Edge case: newBalance == currentBalance

- **Frontend handles this** — the frontend will validate and prevent sending a request when `newBalance` equals the current balance. No special backend handling needed.

## Frontend implementation plan

Once the endpoint is ready:

1. **AccountClient** — add `adjustBalance(accountId: number, data: AdjustBalanceDto): Promise<TransactionDTO>` method
2. **New action on AccountCard** — "Adjust balance" menu item
3. **AdjustBalanceDialog** — Form with:
   - Current balance displayed (read-only)
   - New balance input (number field)
   - Optional description field (text input)
   - Client-side validation: prevent submit if `newBalance == currentBalance`
4. **Mutation hook** — `useAdjustBalanceMutation` → calls endpoint, invalidates account + transaction queries
5. **Offline support** — Update balance in IndexedDB locally, queue sync operation for when back online
6. **Home CurrentBalanceCard** — Add adjust action there too
7. **Translations** — Add keys for the new action/dialog in `en` and `es`
8. **Update `initial` → `auto`** — Frontend must update all references to the renamed flag
